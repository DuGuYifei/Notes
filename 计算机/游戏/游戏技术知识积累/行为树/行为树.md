# 行为树

行为树（Behavior Tree，BT）是游戏 AI 中最常见的**决策与行为组织方式**之一。它把“AI 该做什么”拆成一棵可组合、可复用、可调试的树，广泛用于 NPC 敌人、队友、动物、Boss、任务 AI 等。

---

## 1. 行为树解决了什么问题

传统 if-else / 状态机在规模变大后容易出现：

- 条件分散在各处，难维护
- 状态切换路径爆炸，改一处容易影响全局
- 复用困难（不同 AI 的共同行为难共享）
- 策划/设计调参与可视化调试不友好

行为树的核心优势：

- **层次化**：从“高层策略”到“底层动作”逐层细化
- **组合化**：通过组合节点表达复杂决策，不必写成巨型分支
- **可复用**：子树可复用，行为模块化
- **可中断**：支持高优先级行为随时打断低优先级行为
- **可视化**：天然适合编辑器图形化展示与运行时调试

## 2. 核心概念

### 2.1 节点返回值（状态）

行为树节点每次执行通常返回三种状态：

- **Success（成功）**：本节点完成且成功
- **Failure（失败）**：本节点完成但失败
- **Running（进行中）**：本节点还没完成，下帧继续

这三个返回值是整棵树流转的基础。

### 2.2 Tick（轮询/驱动）

AI 系统每帧（或固定频率）会对行为树进行一次 Tick：

- 从根节点开始评估
- 根据组合节点规则向下执行
- 直到某处返回 Running / Success / Failure

> 常见做法：并非所有节点每帧都从头“重算”。很多实现会保留运行路径，减少开销。

黑板是行为树共享数据容器，存放决策上下文，例如：

黑板里的每一项数据称为**黑板键（Blackboard Key）**。可以把它理解成“键值对里的 Key”，例如：

- 键名：`TargetActor`，值：当前锁定目标对象
- 键名：`DistanceToTarget`，值：与目标的实时距离
- 键名：`CanSeeTarget`，值：是否可见（布尔值）

行为树节点不直接互相传参，而是通过读写这些黑板键协作。

- `TargetActor`
- `LastKnownTargetPos`
- `HasAmmo`
- `DistanceToTarget`
- `PatrolPoints`

节点通过读写黑板进行协作（条件判断、动作输入输出）。

## 3. 节点类型（最常见）

通常分为 4 大类：

## 3.1 组合节点（Composite）

决定“多个子节点怎么执行”。

### Selector（选择器，或 Priority）

- 从左到右尝试子节点
- 子节点返回 Success：Selector 立刻 Success
- 子节点返回 Running：Selector 返回 Running（并记住当前子节点）
- 所有子节点都 Failure：Selector 才 Failure

用途：表示“做 A，不行做 B，再不行做 C”。

### Sequence（顺序器）

- 从左到右执行子节点
- 子节点返回 Failure：Sequence 立刻 Failure
- 子节点返回 Running：Sequence 返回 Running
- 所有子节点都 Success：Sequence 才 Success

用途：表示“先 A 再 B 再 C，任何一步失败则整套失败”。

### Parallel（并行，部分引擎提供）

- 同时驱动多个子节点
- 根据策略决定何时成功/失败（如“主任务完成即成功”“任一失败则失败”等）

用途：例如“移动到目标 + 同时播放瞄准动画 + 持续观察威胁”。

---

## 3.2 装饰节点（Decorator）

对某个子树“加条件或改语义”。

常见 Decorator：

- **Condition**：条件检查（例如 `CanSeeTarget == true`）
- **Inverter**：反转 Success/Failure
- **Cooldown**：冷却时间内不让子树重复触发
- **Repeat / Retry**：重复执行或失败重试
- **Abort / Observer**：当条件变化时中断当前分支，切换到更高优先级分支

Decorator 是行为树灵活性的关键。

---

## 3.3 叶子节点（Leaf）

真正做事或做判断的最小单元。

### Condition（条件叶）

- 只判断，不产生复杂副作用
- 通常快速返回 Success/Failure

### Action（动作叶）

执行实际动作，例如：

- 移动到点位
- 播放动画
- 开火
- 搜索掩体
- 调用导航/技能/感知系统

Action 经常会返回 Running（比如移动需要多帧）。

---

## 3.4 服务节点（Service，部分实现如 UE）

以固定频率在某个子树上执行后台逻辑，例如每 0.2 秒：

- 更新目标距离
- 刷新威胁评分
- 采样环境信息

优点：把“周期更新”从动作里拆出来，减少重复计算。

---

## 4. 执行语义（非常重要）

理解行为树，关键是理解“Running + 中断 + 记忆”。

### 4.1 Running 的意义

当 Action 返回 Running，说明任务跨帧执行：

- 本帧发起路径跟随
- 后续帧持续检测是否到达
- 到达后返回 Success；失败返回 Failure

Running 除了表示“还没做完”，还隐含一个关键作用：**让树记住当前做到哪一步**。

- 例如 Sequence 执行到第 3 个子节点 `MoveToTarget` 返回 Running。
- 下一帧如果该 Sequence 是“有记忆”的，就会直接从第 3 个子节点继续，而不是回到第 1 个条件重跑。
- 这样可以避免反复初始化动作（重复下发 Move 命令）、减少计算开销、让行为更稳定。

如果实现是“无记忆”，则每帧都可能从头评估：优点是更敏感，缺点是更容易抖动与重复开销。

### 4.2 条件中断（Abort）

常见需求：

- 正在巡逻时突然看到敌人，立刻切战斗
- 正在追击时血量过低，立刻找掩体

这通常通过 Decorator 监听黑板值变化并触发中断来实现。

### 4.3 记忆/非记忆节点

一些实现区分：

- **有记忆（Memorized）**：下帧继续上次 Running 的子节点
- **无记忆**：每帧从第一个子节点重新评估

这会直接影响性能和行为表现。

## 5. 一个实战例子：敌人 AI（巡逻 / 追击 / 攻击）

目标：高优先级先攻击，其次追击，再次巡逻。

```text
Root
└── Selector（高优先级在左）
	 ├── Sequence（攻击分支）
	 │   ├── Cond: CanSeeTarget
	 │   ├── Cond: InAttackRange
	 │   └── Act : AttackTarget
	 ├── Sequence（追击分支）
	 │   ├── Cond: CanSeeTarget
	 │   └── Act : MoveToTarget
	 └── Sequence（巡逻分支）
		  ├── Act : FindNextPatrolPoint
		  └── Act : MoveToPatrolPoint
```

执行特征：

- 看得到且够近 -> 攻击
- 看得到但不够近 -> 追击
- 看不到 -> 巡逻

如果加入 Abort，当 `CanSeeTarget` 从 false 变 true 时，AI 可从巡逻即时切到追击/攻击。

为了直观理解 Running + 记忆，这个例子可以按时间看：

- **T0**：看不到敌人，进入巡逻分支，`MoveToPatrolPoint` 返回 Running。
- **T1**：有记忆时，巡逻分支会继续上次的移动过程，而不是每帧重新 `FindNextPatrolPoint`。
- **T2**：突然看到敌人，`CanSeeTarget=true` 触发 Abort，中断巡逻 Running。
- **T3**：切到追击分支，`MoveToTarget` 返回 Running，持续追击。
- **T4**：进入攻击距离，追击结束（Success），转入攻击分支执行 `AttackTarget`。

这就是“Running 让动作可跨帧持续，Abort 让高优先级决策可即时抢占”。

## 6. 与其他 AI 架构对比

### 6.1 行为树 vs 有限状态机（FSM）

- FSM：简单直接，适合小规模、离散状态少的角色
- BT：更适合复杂层级与可复用行为
- 当状态与转移增多时，FSM 容易“状态爆炸”

### 6.2 行为树 vs GOAP

- GOAP（目标导向规划）擅长动态规划行动序列
- BT 擅长稳定、可控、易调试的手工决策结构
- 常见混合：GOAP 负责“选计划”，BT 负责“执行计划”

### 6.3 行为树 vs Utility AI

- Utility AI 通过打分选择当前最佳行为
- BT 通过树结构和条件控制优先级与流程
- 常见混合：Utility 选“策略分支”，BT 管“策略内执行细节”

## 7. 工程落地建议

### 7.1 黑板键设计规范

- 命名统一（如 `HasTarget`, `TargetActor`, `TargetPos`）
- 区分只读与可写来源，避免节点互相覆盖
- 对对象引用设置失效检查（目标死亡/销毁）

### 7.2 Action 设计原则

- 单一职责：一个 Action 只做一件事
- 支持取消：被中断时能安全清理（停移动、停动画、回收句柄）
- 明确成功/失败条件，避免永远 Running

### 7.3 性能优化

- 降低全树每帧重算频率（可分层 Tick）
- Service 用合理周期，不要每帧做昂贵查询
- 感知结果事件驱动优先于高频轮询

### 7.4 调试手段

- 可视化显示当前运行路径（高亮 Running 节点）
- 记录关键黑板键变化日志
- 为 Action 增加执行计时和失败原因统计

## 8. 常见问题与坑

1. **节点副作用太多**
	- 条件节点里做了写操作，导致行为难以预测。

2. **返回值语义混乱**
	- 有的 Action 失败条件不明确，导致 Selector/Sequence 逻辑失真。

3. **中断风暴**
	- 条件抖动导致频繁切分支，出现 AI 抽搐。
	- 可通过滞后阈值、冷却 Decorator、稳定时间窗口解决。

4. **黑板脏数据**
	- 目标无效后未清空，AI 持续追无效对象。

5. **树过深、过大**
	- 阅读和维护困难。
	- 应拆分子树、抽公共行为模块。

---

## 9. 适用场景与不适用场景

适用：

- 需要可视化编辑与调试
- 行为层次明显、可模块化
- 角色类型多、行为复用需求高

不太适用（或需混合方案）：

- 高度动态的长链规划问题（可引入 GOAP）
- 纯数值决策竞争（可叠加 Utility 打分）

---

## 10. 一句话总结

行为树本质上是：**用树形组合结构管理“条件判断 + 动作执行 + 中断切换”的实时决策系统**。它兼顾可读性、可维护性和工程可落地性，是现代游戏 AI 的基础技术之一。
